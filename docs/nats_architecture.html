NATS Architecture - PUDA Comms

Introduction

What is NATS?

NATS is a lightweight, high-performance messaging system that enables applications to communicate with each other. Think of it as a message bus or postal service for your applications:

  • Publishers send messages to specific subjects (like mailing addresses)
  • Subscribers listen to subjects and receive messages
  • The NATS server routes messages from publishers to subscribers

What is a Microservices Architecture?

In a microservices architecture, a large application is broken down into smaller, independent services that communicate over a network. Instead of one monolithic application, you have:

  • Multiple services that can be developed, deployed, and scaled independently
  • Loose coupling - services don't need to know about each other's internal implementation
  • Message-based communication - services communicate by sending messages rather than direct function calls

Why Use NATS for Machine Communication?

This system uses NATS because:

  • Decoupling: Command senders don't need to know which machines are online or where they are
  • Reliability: Messages are persisted and guaranteed to be delivered (using JetStream)
  • Scalability: Can handle many machines and many command senders simultaneously
  • Flexibility: New services can subscribe to telemetry or events without modifying existing code
  • Resilience: If a machine disconnects, messages wait in queues until it reconnects

Core Concepts

Subjects (Message Addresses)

A subject is like an address or topic name. Messages are published to subjects, and subscribers listen to subjects.

Subject Pattern: puda.{machine_id}.{category}.{sub_category}

Example: puda.first-machine.tlm.heartbeat

  • puda - namespace (all PUDA messages)
  • first-machine - which machine
  • tlm - category (telemetry)
  • heartbeat - specific type

Core NATS vs JetStream

This system uses two NATS features:

Feature Core NATS (Fire-and-Forget) JetStream (Persistent)
Persistence No - messages are not stored Yes - messages are stored in streams
Reliability Best effort - if no subscriber, message is lost Guaranteed delivery
Best For Telemetry, events, real-time data Commands, responses, critical data
Performance Very fast, lightweight Slower due to persistence overhead

Streams and WorkQueue Retention

A stream is like a message queue that stores messages. WorkQueue Retention means:

  • Each message is delivered to exactly one subscriber
  • Once processed and acknowledged, the message is removed
  • Ensures exactly-once processing - no duplicate work
  • Perfect for commands that should only be executed once
Subject Pattern: puda.{machine_id}.{category}.{sub_category}

Message Types

Telemetry (Core NATS - Fire-and-Forget)

What is it? Real-time status updates from machines (like sensor readings).

Why Fire-and-Forget? Telemetry is high-frequency (sent many times per second), real-time (old data isn't useful), and lightweight (no persistence overhead). If a subscriber misses a message, the next one comes soon.

Subject Description Transport
puda.{machine_id}.tlm.heartbeat Periodic heartbeat with timestamp (machine is alive) Core NATS
puda.{machine_id}.tlm.pos Real-time position coordinates (where the machine is) Core NATS
puda.{machine_id}.tlm.health System health vitals (CPU, memory, temperature) Core NATS

Commands (JetStream - Exactly-Once Delivery)

What is it? Instructions sent to machines to perform actions.

Why JetStream? Commands must be delivered (can't lose them), must be processed once (don't want duplicates), and must persist (survive machine disconnections).

Two Types:

  • Queue Commands - Processed sequentially, one at a time, in order (e.g., "Move to X, then Y, then Z")
  • Immediate Commands - Priority actions that can interrupt queue processing (e.g., "Stop everything immediately!")
Stream Subject Retention Usage
COMMAND_QUEUE puda.{machine_id}.cmd.queue WorkQueue Execute commands (sequential processing, one at a time)
COMMAND_IMMEDIATE puda.{machine_id}.cmd.immediate WorkQueue Pause, Resume, Cancel, and other immediate actions (can interrupt queue)

Command Responses (JetStream - Exactly-Once Delivery)

What is it? Confirmation messages sent back from machines after processing commands.

Why Separate Response Streams? Responses must be delivered to the command sender, must be processed once (don't want duplicate acknowledgments), and must persist (survive network issues).

Note: Both response streams (RESPONSE_QUEUE and RESPONSE_IMMEDIATE) use WorkQueue retention to ensure exactly-once processing of responses.

Stream Subject Retention Usage
RESPONSE_QUEUE puda.{machine_id}.cmd.response.queue WorkQueue Responses to queue commands (contains: success/failure status, completion time, error messages)
RESPONSE_IMMEDIATE puda.{machine_id}.cmd.response.immediate WorkQueue Responses to immediate commands (confirmation that pause/resume/cancel was executed)

Events (Core NATS - Fire-and-Forget)

What is it? Notifications about things that happened (like log entries or alerts).

Why Fire-and-Forget? Events are informational (not critical for operation), high volume (many events can be generated), and real-time (old events aren't as useful). Multiple services can listen without coordination.

Use Cases: Logging service subscribes to all *.evt.log subjects, alerting service subscribes to all *.evt.alert subjects, media storage service subscribes to all *.evt.media subjects.

Subject Description Transport
puda.{machine_id}.evt.log Log events (informational messages) Core NATS
puda.{machine_id}.evt.alert Alert events for critical issues (warnings, errors) Core NATS
puda.{machine_id}.evt.media Media events (images, videos captured) Core NATS

Status (JetStream KV Store)

What is it? Current state of each machine, stored as key-value pairs.

Why KV Store? Fast lookups ("What's the status of machine X?" - instant answer), persistent (status survives server restarts), and simple (just key-value pairs, no complex queries needed).

Example Value:

{
  "state": "busy",
  "run_id": "run-12345",
  "timestamp": "2024-01-15T10:30:00Z"
}

Use Cases: Dashboard shows current status of all machines, monitoring service checks if machines are stuck, load balancer routes commands based on machine availability.

Bucket Key Description
MACHINE_STATE_{machine_id} {machine_id} Current state (idle/busy/error), run_id, timestamp

Key Features

  • Exactly-Once Delivery: All commands and responses use JetStream with WorkQueue retention
  • Four Streams Per Machine: Two command streams (queue and immediate) and two response streams (queue and immediate) ensure reliable command-response communication
  • Keep-Alive: Messages send periodic in_progress() signals during long-running operations (every 25 seconds)
  • Cancellation: Commands can be cancelled by run_id
  • Pause/Resume: Queue processing can be paused and resumed
  • Auto-Reconnection: Client automatically reconnects and resubscribes on connection loss
  • Status Tracking: Machine state stored in KV store for quick lookup
  • Durable Subscriptions: Subscriptions survive reconnections
  • Fire-and-Forget: Telemetry and events use Core NATS for lightweight, non-persistent messaging
