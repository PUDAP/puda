NATS Architecture - PUDA Comms

Introduction

What is NATS?

NATS is a lightweight, high-performance messaging system that enables applications to communicate with each other. Think of it as a message bus or postal service for your applications:

  • Publishers send messages to specific subjects (like mailing addresses)
  • Subscribers listen to subjects and receive messages
  • The NATS server routes messages from publishers to subscribers

What is a Microservices Architecture?

In a microservices architecture, a large application is broken down into smaller, independent services that communicate over a network. Instead of one monolithic application, you have:

  • Multiple services that can be developed, deployed, and scaled independently
  • Loose coupling - services don't need to know about each other's internal implementation
  • Message-based communication - services communicate by sending messages rather than direct function calls

Why Use NATS for Machine Communication?

This system uses NATS because:

  • Decoupling: Command senders don't need to know which machines are online or where they are
  • Reliability: Messages are persisted and guaranteed to be delivered (using JetStream)
  • Scalability: Can handle many machines and many command senders simultaneously
  • Flexibility: New services can subscribe to telemetry or events without modifying existing code
  • Resilience: If a machine disconnects, messages wait in queues until it reconnects

Core Concepts

Subjects (Message Addresses)

A subject is like an address or topic name. Messages are published to subjects, and subscribers listen to subjects.

Subject Pattern: puda.{machine_id}.{category}.{sub_category}

Note: Machine IDs with dots (`.`) are automatically replaced with dashes (`-`) in subject names.

Example: puda.first-machine.tlm.heartbeat

  • puda - namespace (all PUDA messages)
  • first-machine - which machine (dots replaced with dashes)
  • tlm - category (telemetry)
  • heartbeat - specific type

Core NATS vs JetStream

This system uses two NATS features:

Feature Core NATS (Fire-and-Forget) JetStream (Persistent)
Persistence No - messages are not stored Yes - messages are stored in streams
Reliability Best effort - if no subscriber, message is lost Guaranteed delivery
Best For Telemetry, events, real-time data Commands, responses, critical data
Performance Very fast, lightweight Slower due to persistence overhead

Streams and Retention Policies

A stream is like a message queue that stores messages. Different retention policies control how messages are stored and delivered:

WorkQueue Retention

  • Each message is delivered to exactly one subscriber
  • Once processed and acknowledged, the message is removed
  • Ensures exactly-once processing - no duplicate work
  • Perfect for commands that should only be executed once
  • Used for: Command streams (queue and immediate)

Interest Retention

  • Messages are kept as long as there are active consumers
  • Once all consumers have processed a message, it can be removed
  • Multiple consumers can process the same message
  • Perfect for responses where multiple services might need to see them
  • Used for: Response streams (queue and immediate)

Pull vs Push Consumers

This system uses two types of JetStream consumers:

Pull Consumer (Queue Commands)

  • Machine-initiated: Machine actively pulls messages when ready
  • Efficient: Machine controls when to fetch messages
  • Best for: Queue commands that need sequential processing

Push Consumer (Immediate Commands)

  • Server-initiated: NATS server pushes messages to machine immediately
  • Fast: Messages delivered as soon as they arrive
  • Best for: Immediate commands (pause, resume, cancel) that need instant delivery
Subject Pattern: puda.{machine_id}.{category}.{sub_category}
Note: Machine IDs with dots (`.`) are automatically replaced with dashes (`-`) in subject names.

Message Structure

All NATS messages follow a structured format:

{
  "header": {
    "message_type": "command" | "response",
    "version": "1.0",
    "timestamp": "2024-01-15T10:30:00Z",
    "user_id": "user-123",
    "username": "john.doe",
    "machine_id": "first-machine",
    "run_id": "run-12345"
  },
  "command": {
    "name": "move",
    "params": {...},
    "step_number": 1,
    "version": "1.0"
  },
  "response": {
    "status": "success" | "error",
    "completed_at": "2024-01-15T10:30:05Z",
    "code": "EXECUTION_ERROR" | null,
    "message": "Error description" | null,
    "data": {...}
  }
}

Key Fields:

  • run_id: Unique identifier for a workflow/sequence of commands. Required for queue commands.
  • step_number: Sequential number for commands in a run (0, 1, 2, ...)
  • message_type: command for requests, response for replies

Run Management

The system uses run_id to group related commands into a workflow:

  1. START Command: Begins a new run (immediate command)
    • Sets the active run_id on the machine
    • Machine validates all subsequent queue commands match this run_id
    • If a command has a different run_id, it's rejected
  2. Queue Commands: Execute sequentially within a run
    • Must have a run_id that matches the active run
    • Processed one at a time, in order
    • Each command has a step_number (1, 2, 3, ...)
  3. COMPLETE Command: Ends the run (immediate command)
    • Clears the active run_id
    • Machine returns to idle state
  4. CANCEL Command: Cancels the current run (immediate command)
    • Clears the active run_id
    • Stops processing queue commands for that run

Benefits: Prevents commands from different workflows from interfering, ensures commands execute in the correct sequence, allows cancellation of entire workflows.

Message Types

Telemetry (Core NATS - Fire-and-Forget)

What is it? Real-time status updates from machines (like sensor readings).

Why Fire-and-Forget? Telemetry is high-frequency (sent many times per second), real-time (old data isn't useful), and lightweight (no persistence overhead). If a subscriber misses a message, the next one comes soon.

Subject Description Transport
puda.{machine_id}.tlm.heartbeat Periodic heartbeat with timestamp (machine is alive) Core NATS
puda.{machine_id}.tlm.pos Real-time position coordinates (where the machine is) Core NATS
puda.{machine_id}.tlm.health System health vitals (CPU, memory, temperature) Core NATS

Commands (JetStream - Exactly-Once Delivery)

What is it? Instructions sent to machines to perform actions.

Why JetStream? Commands must be delivered (can't lose them), must be processed once (don't want duplicates), and must persist (survive machine disconnections).

Two Types:

  • Queue Commands - Processed sequentially, one at a time, in order (e.g., "Move to X, then Y, then Z"). Uses pull consumer (machine pulls messages when ready). Requires run_id that matches active run.
  • Immediate Commands - Priority actions that can interrupt queue processing (e.g., "Stop everything immediately!"). Uses push consumer (NATS pushes messages immediately). Built-in commands: START, COMPLETE, PAUSE, RESUME, CANCEL.
Stream Subject Retention Consumer Type Usage
COMMAND_QUEUE puda.{machine_id}.cmd.queue WorkQueue Pull Execute commands (sequential processing, one at a time). Requires run_id matching active run.
COMMAND_IMMEDIATE puda.{machine_id}.cmd.immediate WorkQueue Push START, COMPLETE, PAUSE, RESUME, CANCEL, and other immediate actions (can interrupt queue)

Command Responses (JetStream - Interest Retention)

What is it? Confirmation messages sent back from machines after processing commands.

Why Interest Retention? Responses must be delivered to the command sender, multiple services might need to see responses (logging, monitoring), messages are kept as long as there are active consumers, and messages are automatically cleaned up when no longer needed.

Note: Both response streams (RESPONSE_QUEUE and RESPONSE_IMMEDIATE) use Interest retention to allow multiple consumers to process responses.

Stream Subject Retention Usage
RESPONSE_QUEUE puda.{machine_id}.cmd.response.queue Interest Responses to queue commands (contains: success/failure status, completion time, error messages)
RESPONSE_IMMEDIATE puda.{machine_id}.cmd.response.immediate Interest Responses to immediate commands (confirmation that pause/resume/cancel/start/complete was executed)

Events (Core NATS - Fire-and-Forget)

What is it? Notifications about things that happened (like log entries or alerts).

Why Fire-and-Forget? Events are informational (not critical for operation), high volume (many events can be generated), and real-time (old events aren't as useful). Multiple services can listen without coordination.

Use Cases: Logging service subscribes to all *.evt.log subjects, alerting service subscribes to all *.evt.alert subjects, media storage service subscribes to all *.evt.media subjects.

Subject Description Transport
puda.{machine_id}.evt.log Log events (informational messages) Core NATS
puda.{machine_id}.evt.alert Alert events for critical issues (warnings, errors) Core NATS
puda.{machine_id}.evt.media Media events (images, videos captured) Core NATS

Status (JetStream KV Store)

What is it? Current state of each machine, stored as key-value pairs.

Why KV Store? Fast lookups ("What's the status of machine X?" - instant answer), persistent (status survives server restarts), and simple (just key-value pairs, no complex queries needed).

Example Value:

{
  "state": "active",
  "run_id": "run-12345",
  "timestamp": "2024-01-15T10:30:00Z"
}

Use Cases: Dashboard shows current status of all machines, monitoring service checks if machines are stuck, load balancer routes commands based on machine availability.

Bucket Key Description
MACHINE_STATE_{machine_id} {machine_id} Current state (idle/active/paused/error), run_id, timestamp

Key Features

  • Exactly-Once Delivery: All commands use JetStream with WorkQueue retention
  • Four Streams Per Machine: Two command streams (queue and immediate) and two response streams (queue and immediate) ensure reliable command-response communication
  • Keep-Alive: Messages send periodic in_progress() signals during long-running operations (every 25 seconds). Resets the redelivery timer so messages aren't redelivered.
  • Run Management: Commands are grouped by run_id into workflows. START command begins a run, queue commands must match active run_id, COMPLETE command ends a run, CANCEL command cancels a run.
  • Pause/Resume: Queue processing can be paused and resumed. PAUSE and RESUME are immediate commands.
  • Auto-Reconnection: Client automatically reconnects and resubscribes on connection loss. Durable consumers ensure messages aren't lost.
  • Status Tracking: Machine state stored in KV store for quick lookup
  • Pull vs Push Consumers: Pull consumer for queue commands (machine controls when to fetch), push consumer for immediate commands (NATS delivers immediately)
  • Message Structure: Standardized message format with header, command, and response. Type-safe with Pydantic models.
  • Fire-and-Forget: Telemetry and events use Core NATS for lightweight, non-persistent messaging
  • Different Retention Policies: WorkQueue retention for commands (exactly-once processing), Interest retention for responses (multiple consumers can process)
